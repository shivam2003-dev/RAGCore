# Web Search and Council Mode

This document describes the optional Ask modes that extend normal internal RAG. They are disabled unless configured, so the UI must not imply live internet access or multi-model synthesis when provider credentials are absent.

## Source Modes

Ask supports three source modes:

- `knowledge`: the default. Searches local synced Jira, Confluence, and uploaded-document chunks only.
- `web`: calls the configured web-search provider, stores normalized snippets as local `Web Search` documents, embeds them, and cites those local chunks.
- `blended`: runs internal retrieval and web search, then sends the combined source list to the answer model.

Normal knowledge retrieval excludes the generated `Web Search` knowledge base. This prevents stale web snippets from silently affecting internal-only answers.

## Web Search Providers

Configure one provider:

```bash
WEB_SEARCH_PROVIDER=brave       # brave | tavily | searxng
WEB_SEARCH_API_KEY=             # required for brave and tavily
WEB_SEARCH_BASE_URL=            # optional for brave/tavily, required for searxng
WEB_SEARCH_DEFAULT_KB_NAME=Web Search
WEB_SEARCH_TOP_K=5
```

Provider notes:

- Brave uses the Web Search API endpoint with `X-Subscription-Token`.
- Tavily uses the `/search` endpoint with bearer auth.
- SearXNG uses a JSON-enabled `/search?q=...&format=json` instance.

The backend exposes `GET /api/v1/web-search/status` so the UI can disable Web and Both when search is not configured.

## Organization Space Radar

The Ask focus page includes SRE, DevOps, Dev, HR, and Custom spaces. Each space has radar prompts for current field awareness, such as:

- security releases, CVEs, zero-days, and exploit writeups
- conferences, talks, books, and long-form articles
- major research work and implementation guidance
- operational alerts and best-practice changes in that field

When a user clicks a radar prompt and web search is configured, the UI switches the source mode to `blended` so the answer can combine synced Jira/Confluence/uploaded documents with fresh web snippets. If web search is not configured, the UI keeps the request on internal knowledge and shows the provider status reason.

## Role Spaces and Custom Roles

Every Ask request can include an optional assistant role:

- `assistant_role`: display name such as `SRE Space` or `Security Architect`.
- `assistant_role_prompt`: background role prompt generated by the selected org space or custom role builder.

The backend injects this as an `<assistant_role>` block inside the system prompt after the non-negotiable source and security hierarchy. Role prompts can tune persona, workflow, source preference, and answer format. They cannot override RBAC, citation, source-grounding, secret handling, PII, or prompt-injection protections.

Built-in spaces are static product defaults:

- SRE Space
- DevOps Space
- Dev Space
- HR Space

Custom roles are generated by `POST /api/v1/chat/roles/generate`. The UI asks for role name, goal, source focus, and output style, then sends those answers to the configured LLM. The generated prompt is stored in browser local storage and sent with future Ask requests from that browser session.

## LLM Council Mode

Fast mode streams one configured LLM answer. Council mode fans out to configured OpenAI-compatible model ids, then asks a chair model to synthesize one final answer. Candidate answers are treated as advisory analysis only; source chunks remain the only evidence.

```bash
LLM_COUNCIL_ENABLED=true
LLM_COUNCIL_AVAILABLE_MODELS=model-a,model-b,model-c
LLM_COUNCIL_MODELS=model-a,model-b       # optional startup default
LLM_COUNCIL_CHAIR_MODEL=model-a          # optional startup default
LLM_COUNCIL_API_KEY=            # optional if OPENROUTER_API_KEY or OPENAI_API_KEY is set
LLM_COUNCIL_BASE_URL=https://openrouter.ai/api/v1
LLM_COUNCIL_MAX_MODELS=3
LLM_COUNCIL_TIMEOUT_SECONDS=120
```

Runtime behavior:

- The UI reads `GET /api/v1/chat/capabilities`.
- The UI lets the user choose two or three models from `council_available_models`.
- The user chooses one selected model as chair. By default the app prefers `anthropic/claude-haiku-4.5`, then `anthropic/claude-sonnet-4.5`, then the first selected model.
- The chat request sends `answer_mode=council`, `council_models`, and `council_chair_model`.
- The backend validates the selected models against `LLM_COUNCIL_AVAILABLE_MODELS`; provider keys stay server-side.
- The backend returns `answer_mode`, `source_mode`, and `model` in the terminal `done` SSE event.
- Persisted council answers use a model label like `llm-council:<chair-model>`.
- The answer card displays `Council - <chair-model>` so the user can verify that Council mode actually ran.

Council remains disabled unless the backend has `LLM_COUNCIL_ENABLED=true`, an OpenAI-compatible API key, and at least two selectable models. This avoids fake council behavior in production.

### Current local configuration

The local development environment is configured through gitignored `backend/.env`. It has:

- `LLM_COUNCIL_ENABLED` set.
- `OPENROUTER_API_KEY` set, which is reused by Council when `LLM_COUNCIL_API_KEY` is empty.
- `LLM_COUNCIL_AVAILABLE_MODELS` controls the dropdown choices.
- `LLM_COUNCIL_MODELS` is optional; when present it provides the startup default selection.

When `LLM_COUNCIL_MODELS` is empty and the LLM provider is OpenRouter or OpenAI, the capabilities endpoint still exposes the available dropdown list. The UI sends the selected two or three model ids with each Council request, so the actual council membership is user-configured per browser session.

Example OpenRouter setup:

```bash
LLM_PROVIDER=openrouter
LLM_MODEL=anthropic/claude-haiku-4.5
LLM_BASE_URL=https://openrouter.ai/api/v1
OPENROUTER_API_KEY=...

LLM_COUNCIL_ENABLED=true
LLM_COUNCIL_AVAILABLE_MODELS=anthropic/claude-haiku-4.5,openai/gpt-4.1-mini,google/gemini-2.5-flash
LLM_COUNCIL_MODELS=anthropic/claude-haiku-4.5,openai/gpt-4.1-mini
LLM_COUNCIL_CHAIR_MODEL=anthropic/claude-haiku-4.5
LLM_COUNCIL_BASE_URL=https://openrouter.ai/api/v1
LLM_COUNCIL_MAX_MODELS=3
```

Do not commit real provider keys. Keep them in `backend/.env` or the production secret manager.

## Citation Behavior

Web snippets become local `DocumentVersion` and `Chunk` rows before answer generation. That means answer markers such as `[1]` still map back to persisted chunk ids and remain compatible with existing citation storage.
